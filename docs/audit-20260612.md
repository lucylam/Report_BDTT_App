## 3. Thay đổi đã thực hiện

1. **`lib/api/security.ts`** (mới): helper `isAllowedRequestOrigin` kiểm tra `Sec-Fetch-Site` + `Origin` vs `Host`, áp dụng cho `POST /api/progress/submit`, `POST /api/tasks/import`, `POST /api/google-sheets/sync-data` → trả 403 nếu cross-site.
2. **`app/worker/page.tsx`**: thêm `syncOfflineQueue()` — khi có mạng lại (event `online` hoặc khi còn hàng đợi), gửi từng bản ghi trong `offlineQueue` lên `/api/progress/submit` rồi mới `flushQueue()`; nếu lỗi giữa chừng thì giữ hàng đợi để retry (server upsert idempotent). Khi submit online bị lỗi mạng (`TypeError`), bản ghi được xếp vào hàng đợi và badge hiển thị "Lưu offline" thay vì "Lỗi lưu".
3. **`lib/storage.ts`**: `changeAccountPassword` từ chối mật khẩu mặc định `123456` và mật khẩu trùng mật khẩu hiện tại.
4. **`app/change-password/page.tsx`**: hiển thị quy tắc mật khẩu ngay trên form.
5. **`vitest` + `tests/`**: 22 unit tests cho `lib/progress`, `lib/permissions`, `lib/date`, password policy. Script: `npm test`.

## 4. Rủi ro còn lại — cần quyết định của chủ dự án

1. **Kiến trúc xác thực client-side**: tài khoản + mật khẩu plaintext nằm trong localStorage; role kiểm tra phía client; API dùng service-role key. Origin check mới chỉ chặn CSRF từ trình duyệt — **không chặn được kẻ gọi API trực tiếp**. Khuyến nghị: chuyển sang Supabase Auth (đã có sẵn `@supabase/supabase-js` và bảng `profiles`), API verify JWT thay vì tin `username` trong body.
2. **`DEFAULT_REPORT_DATE = "2025-08-22"`**: nếu chiến dịch BDTT 2026 báo cáo theo ngày thực tế, cần đổi thành ngày hiện tại (giờ VN) hoặc chọn từ `REPORT_DATES` của đợt 2026. Chưa tự đổi vì ảnh hưởng dữ liệu đã ghi trong DB.
3. **Demo data seed tự động**: cân nhắc tách chế độ demo khỏi production bằng env flag.
